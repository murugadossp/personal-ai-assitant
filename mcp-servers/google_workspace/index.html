🔐 Multi-User OAuth Flow

First-Time Authorization

1

Agent calls MCP tool

Tool returns "User not authorized" + auth URL

2

User opens auth URL

Server generates state token, redirects to Google

3

Google consent screen

"Allow Calendar, Gmail, Drive access?"

4

User clicks Allow

Google sends code to /auth/callback

5

Tokens stored

access_token + refresh_token saved per user

Subsequent API Calls

1

Agent calls MCP tool

e.g. calendar_list_events(user_id="user1")

2

Token loaded from store

Local file (.tokens/) or Firestore

3

Auto-refresh if expired

Uses refresh_token → new access_token

4

Google API called

Bearer token in Authorization header

5

Data returned via MCP

JSON result sent back to agent

💾 Token Storage

🗂️ Local (Development)

Per-user JSON files in .tokens/ directory. Simple, no external dependencies. Set TOKEN_BACKEND=local.

.tokens/ ├── user1.json # {token, refresh_token, ...} ├── judge1.json └── demo-user.json

🔥 Firestore (Production)

Google Cloud Firestore collection mcp_user_tokens. Serverless, auto-scales, encrypted at rest. Set TOKEN_BACKEND=firestore.

Firestore: mcp_user_tokens/ ├── user1 # {token, refresh_token, ...} ├── judge1 # {token, refresh_token, ...} └── demo-user # {token, refresh_token, ...}

🚀 Quick Start
1

Activate venv & start server

source .venv/bin/activate MCP_TRANSPORT=streamable-http python server.py
2

Authorize your account

open http://localhost:8080/auth?user_id=your-name
3

Test with curl

curl http://localhost:8080/health curl http://localhost:8080/auth/status/your-name
