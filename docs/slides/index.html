Brief about the Idea

Natural language over your real Google Workspace

The project is a personal AI assistant that lets you talk to Calendar, Gmail, and Drive in plain English—without juggling separate apps or brittle custom scripts.

A coordinator agent handles routing and light conversation, then hands workspace tasks to a single workspace sub-agent. That agent calls a dedicated Google Workspace MCP server over Streamable HTTP, keeping OAuth, tokens, and API details out of the agent’s core logic.

Built for Google GenAI APAC Hackathon 2026: demonstrate ADK multi-agent patterns, MCP as a clean tool boundary, and production-minded touches (config-driven Gemini models, deployable MCP and assistant services).

The Workspace MCP server exposes tools (calendar_*, gmail_*, drive_*), runs OAuth and token storage, and talks to Google REST APIs. The assistant stays focused on reasoning and tool orchestration—not low-level auth plumbing.

  • Transport: MCP Streamable HTTP between assistant and server.
  • Deploy: assistant and MCP as separate services (e.g. two Cloud Run services) with mcp_settings.json pointing at the live MCP URL.
Opportunities

Differentiation & unique value

How different is it from any of the other existing ideas?

Many demos stop at RAG or read-only Q&A. This assistant acts on your real account—create events, search mail, open Drive—through OAuth-backed tools, not mocked data. It is also not a tangle of product-specific agents: one workspace specialist plus a coordinator keeps ADK wiring simple while still feeling multi-agent.

The MCP server is a real boundary: auth, token storage, and Google API calls live outside the agent core, so the same assistant pattern can evolve (more tools, stricter policies) without rewriting the whole LLM stack.

USP of the proposed solution

  • Unified Workspace UX: one natural-language thread for Calendar, Gmail, and Drive.
  • Standard tool surface: MCP Streamable HTTP — portable, observable, and swappable backends.
  • Production-shaped defaults: split deploy (assistant vs MCP), config-driven Gemini with ordered model fallbacks, and durable token storage on the MCP side when you choose Firestore.
Product

List of features offered by the solution

User-facing

  • One conversation for Calendar, Gmail, and Drive—ask in plain language.
  • Real account actions (list/create events, search and read mail, list and read Drive files)—not mocked data.
  • Per-user OAuth via the MCP server; first-time /auth?user_id=… flow, then the assistant remembers the session context.
  • FastAPI + optional ADK Web UI for chat; works with /api/chat for integrations.

Coordinator agent

  • Intent routing and light chit-chat at the ADK app root.
  • Delegation with transfer_to_agent("workspace_agent") for anything Workspace-shaped.
  • No MCP tools on the coordinator—keeps the root agent small and predictable.

Workspace agent

  • Single specialist sub-agent with the full Google Workspace MCP toolset (no separate Calendar/Gmail/Drive ADK agents).
  • Tool orchestration—picks the right calendar_*, gmail_*, drive_* calls and summarizes results in natural language.
  • Config-driven Gemini per agent block in config.yaml with ordered model fallbacks on quota-style errors.

Workspace MCP server

  • MCP Streamable HTTP endpoint for tools; works with the ADK MCPToolset.
  • OAuth + token lifecycle; local or Firestore token backend for dev vs scaled deploy.
  • Deployable (e.g. Cloud Run) with BASE_URL-aware auth links.
Build & run

From repo to a live Workspace conversation

Local loop

  • Run Workspace MCP with Streamable HTTP (MCP_TRANSPORT=streamable-http).
  • Run the ADK assistant (./scripts/adk_web.sh or python -m app.main); point mcp_settings.json at …/mcp.
  • First-time OAuth via the MCP base URL (/auth?user_id=…), then ask: “What’s on my calendar?”, “Summarize unread mail,” “Find that doc in Drive.”

Production-shaped deploy

  • Two services: MCP on Cloud Run (set BASE_URL to the public URL) + assistant on Cloud Run with the remote MCP URL in config.
  • Models: Gemini IDs and fallback order in config.yaml — no code change when swapping primaries.
  • Tokens: MCP TOKEN_BACKEND=firestore in the provided Dockerfile for durable storage across instances (override only if you mean to).

User Coordinator Workspace MCP APIs

Each hop is one decision boundary: delegate once, then all Workspace tools and OAuth stay behind MCP until a JSON result returns up the chain.

Use case: “What’s on my calendar this week?”

1
User Sends the prompt through the ADK Web UI or POST /api/chat.
2
Coordinator Classifies as Workspace intent → transfer_to_agent("workspace_agent").
3
Workspace agent Chooses calendar_list_events (or equivalent) with date range; calls MCP.
4
Workspace MCP Resolves user_id, loads refresh/access token, invokes Google Calendar API.
5
Google Calendar Returns events JSON to MCP → tool result to the workspace agent.
6
Response path Workspace agent summarizes in natural language → coordinator surfaces the final answer → user sees the reply.
